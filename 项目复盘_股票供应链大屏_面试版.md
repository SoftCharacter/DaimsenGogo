# 1. 项目概述

## 1.1 项目背景与业务痛点

在股票研究和产业链分析场景中，很多用户并不是只关心单只股票，而是关心一个产业主题背后的完整供应链结构。例如：

- 苹果手机供应链中有哪些 A 股公司？
- 宁德时代电池产业链上游、中游、下游分别有哪些标的？
- 华为昇腾相关的芯片、服务器、封装、材料、应用生态分别对应哪些上市公司？

传统方式下，用户通常需要在多个系统之间来回切换：

```text
┌────────────────────────────────────────────────────────────┐
│                 传统产业链研究流程的问题                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 信息收集分散                                           │
│     需要搜索新闻、研报、公司公告、行业文章和股票软件。        │
│                                                            │
│  2. 产业链结构难以沉淀                                     │
│     查到的公司列表往往是零散文本，缺少上中下游分类。          │
│                                                            │
│  3. 分析结果和行情数据割裂                                 │
│     产业链分析在文档里，实时行情在股票软件里。               │
│                                                            │
│  4. 重复研究成本高                                         │
│     每次研究一个新主题都要重新查资料、整理公司、做分类。      │
│                                                            │
│  5. 展示不直观                                             │
│     面试、汇报或复盘时，很难用一个页面讲清楚主题结构和走势。  │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

这个项目的目标就是把“AI 主题分析”和“股票行情看板”结合起来：用户只需要输入一个产业主题，系统自动生成供应链结构，并把相关股票的实时行情和近期 K 线展示在同一个大屏里。

## 1.2 产品定位

这个项目不是传统股票软件，也不是单纯的大模型聊天工具，而是一个“AI 驱动的主题供应链研究工作台”。

| 产品形态 | 能力边界 | 不足 |
| --- | --- | --- |
| 传统股票软件 | 展示行情、K线、财务数据 | 不负责解释一个产业主题背后的供应链结构 |
| 普通大模型问答 | 能生成产业链分析文本 | 结果不结构化，无法直接和行情数据联动 |
| 研报/资讯工具 | 提供专家内容和行业信息 | 需要人工阅读、归纳和二次整理 |
| 本项目 | AI 生成供应链结构，并联动实时行情与 K线 | 更适合主题研究、演示和个人决策辅助 |

项目的核心定位可以总结为：

> 用 Agent 自动完成产业链拆解，用可视化大屏承接结构化分析结果，用行情数据补足实时观察能力。

## 1.3 项目目标

| 目标 | 设计指标 | 业务意义 |
| --- | --- | --- |
| 降低主题研究门槛 | 用户输入一句话即可生成主题结构 | 把原本需要人工整理的工作自动化 |
| 提升结果可读性 | 供应链分类 + 股票卡片 + K线图 | 让复杂产业关系可以被快速理解 |
| 支持持续观察 | 当前主题股票行情每 30 秒刷新 | 从一次性分析升级为动态看板 |
| 降低外部接口压力 | 指定股票请求、缓存、并发限制 | 避免全量下载行情导致系统卡顿 |
| 方便面试展示 | 前后端分离 + Agent + 数据可视化 | 覆盖产品、前端、后端、AI 和工程优化能力 |

# 2. 技术方案设计

## 2.1 完整技术架构

项目采用前后端分离架构，整体可以分成五层：

```text
┌────────────────────────────────────────────────────────────────────┐
│                         用户交互层                                  │
│  供应链看板 / AI分析页 / 模型配置页 / 历史任务 / 主题侧栏             │
│  用户输入主题、查看分析过程、切换主题、查看行情和K线                  │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
┌──────────────────────────────▼─────────────────────────────────────┐
│                         前端应用层                                  │
│  React + TypeScript + Zustand + React Router + lightweight-charts   │
│  状态管理、路由管理、行情轮询、K线懒加载、卡片和大屏展示              │
└──────────────────────────────┬─────────────────────────────────────┘
                               │ /api
┌──────────────────────────────▼─────────────────────────────────────┐
│                         后端 API 层                                 │
│  FastAPI Routers                                                    │
│  analysis / themes / stocks / config / analysis-tasks               │
│  负责参数校验、接口编排、SSE事件流、主题CRUD和行情代理                │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
┌──────────────────────────────▼─────────────────────────────────────┐
│                         业务服务层                                  │
│  ReAct Agent / 文件服务 / 行情服务 / AkShare适配 / 雪球适配           │
│  负责AI分析、数据格式统一、缓存、并发控制、外部接口降级                │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
┌──────────────────────────────▼─────────────────────────────────────┐
│                         数据源层                                    │
│  大模型API / 雪球实时行情 / AkShare日K / 本地JSON文件                 │
│  提供AI能力、行情能力、K线能力和本地持久化能力                        │
└────────────────────────────────────────────────────────────────────┘
```

这套架构的关键设计思想是“AI 分析结果”和“行情数据”解耦：

- AI 分析负责生成主题结构，结果是相对稳定的。
- 股票行情负责展示实时价格，数据是高频变化的。
- K 线数据负责展示近期走势，数据更新频率低于实时行情。

这样做的好处是主题可以复用，行情可以独立刷新，系统不会因为每次看行情都重新跑一次 AI 分析。

## 2.2 核心处理流程

### 2.2.1 AI 分析流程

```text
┌────────────┐
│ 用户输入主题 │
└─────┬──────┘
      │
      ▼
┌────────────┐
│ 参数校验     │  检查输入是否为空、是否超过长度限制、模型配置是否完整
└─────┬──────┘
      │
      ▼
┌────────────┐
│ 创建任务ID   │  为本次分析生成 task_id，隔离缓存和历史记录
└─────┬──────┘
      │
      ▼
┌────────────┐
│ ReAct分析循环│  大模型进行主题理解、供应链拆解、公司筛选和结构化输出
└─────┬──────┘
      │
      ▼
┌────────────┐
│ SSE流式推送  │  前端实时接收进度、工具结果、最终主题
└─────┬──────┘
      │
      ▼
┌────────────┐
│ 保存主题数据 │  写入 data/themes，供看板复用
└────────────┘
```

这个流程解决了两个问题：

1. 用户不需要等待黑盒式的长请求结束，而是可以看到分析过程。
2. 最终结果被结构化保存，后续看板加载不需要重新调用大模型。

### 2.2.2 看板加载流程

```text
┌──────────────┐
│ 进入供应链看板 │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 加载主题详情   │  从本地主题JSON读取分类和股票列表
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 请求实时行情   │  只请求当前主题下股票，不全量下载A股行情
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 渲染股票卡片   │  展示价格、涨跌幅、成交额和业务说明
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ K线懒加载      │  股票卡片进入视口附近时再请求近一个月日K
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 定时刷新行情   │  每30秒刷新当前主题股票实时行情
└──────────────┘
```

这个流程的核心优化点是：主题先显示，行情异步补充，K 线懒加载，避免用户进入看板时被大量外部请求阻塞。

# 3. 核心模块详细设计

## 3.1 前端看板模块

看板模块主要负责把 AI 生成的主题结构转化为可视化页面。

### 设计要点

1. 使用路由参数表示当前主题。
2. 使用 Zustand 保存当前主题和主题列表。
3. 切换页面时保留当前主题 ID。
4. 切换主题时先判断 `currentTheme.id` 是否和路由 ID 一致，避免旧主题残留。
5. 行情数据用 `code -> quote` 的映射保存，方便股票卡片直接读取。
6. K 线组件使用懒加载，降低首屏压力。

### 解决过的问题

早期版本中，用户从“供应链看板”切到“AI分析”或“模型配置”后，再回到看板会丢失当前主题，需要重新选择。这说明当前主题状态和导航路由没有绑定好。

后续优化为：

- 导航到看板时，如果存在当前主题，则跳转到 `/dashboard/{themeId}`。
- 新建主题成功后，立即写入 `currentTheme`。
- 看板页面只展示和当前路由 ID 匹配的主题。

这样页面切换不会破坏用户已经选中的主题。

## 3.2 AI 分析模块

AI 分析模块的核心是把用户的一句自然语言输入转换成结构化的供应链主题。

### 输入

```json
{
  "query": "宁德时代电池供应链"
}
```

### 输出主题结构

```json
{
  "id": "theme_id",
  "name": "宁德时代电池供应链",
  "description": "主题描述",
  "categories": [
    {
      "id": "upstream_materials",
      "name": "上游材料",
      "order": 1,
      "stocks": [
        {
          "code": "SZ:002460",
          "name": "赣锋锂业",
          "percentage": 80,
          "description": "锂资源与锂盐供应商",
          "category_tag": "上游材料"
        }
      ]
    }
  ]
}
```

### 为什么用结构化输出

如果只让大模型生成一段文本，前端无法稳定渲染，也无法和行情接口联动。结构化输出可以带来三个好处：

1. 前端可以直接按分类渲染。
2. 后端可以根据股票代码请求行情。
3. 历史主题可以保存和复用。

## 3.3 Plan-Execute + ReAct 混合分析设计

供应链分析不是一个简单问答任务。用户输入“宁德时代电池供应链”后，系统不能只依赖大模型一次性生成答案，因为一次性生成会带来三个问题：

1. **过程不可控**：模型可能直接编一个看似合理的股票列表，但没有经过搜索和校验。
2. **任务容易发散**：模型可能从“电池供应链”扩展到泛新能源、整车、光伏等无关方向。
3. **结果难以验证**：如果最终 JSON 里有不存在的股票代码，前端行情接口无法匹配，页面会显示 `--`。

因此项目采用 **Plan-Execute + 局部 ReAct** 的混合架构：

- Plan-Execute 负责全局任务拆解，保证分析按固定 SOP 推进。
- ReAct 负责单个步骤内部的局部探索，允许模型根据工具返回继续搜索、确认或修正。
- 后端主循环负责工具白名单、步骤校验、代码校验和最终结果约束，避免模型自由发挥过度。

### 3.3.1 为什么不用纯 ReAct

纯 ReAct 的优点是灵活，但在供应链分析场景里会过于开放。模型可能不断搜索相似关键词，或者跳过验证直接输出最终答案。

例如，模型在发现“锂电池”后，可能连续搜索“电池”“新能源”“储能”“汽车”，最后混入很多不属于当前主题核心供应链的公司。

所以项目没有让模型自由决定完整流程，而是由后端固定 SOP：

```text
候选发现 → 业务确认 → 候选补全 → 结果分组 → 代码校验 → 最终组装
```

模型只能在当前步骤内行动，不能跳到下一步，也不能提前输出最终答案。

### 3.3.2 为什么不用纯 Plan-Execute

纯 Plan-Execute 的问题是每一步太死板。如果只让模型先生成完整计划，再按计划执行，遇到工具返回为空、公司简称不匹配、股票代码格式不一致时，流程很容易卡住。

供应链分析中有大量局部不确定性：

- 公司简称可能和股票简称不完全一致。
- 一个关键词可能搜出多家公司。
- 有些公司需要查主营业务才能判断是否属于该供应链环节。
- 某个环节候选不足时，需要补充搜索。

因此每个 SOP 步骤内部仍然保留 ReAct：模型可以根据 Observation 决定下一次调用哪个工具、换哪个关键词、是否进入步骤总结。

最终形成的设计是：

```text
全局：Plan-Execute 固定流程，控制方向
局部：ReAct 多轮工具调用，处理不确定性
系统：后端校验和拦截，保证结果合法
```

在这个流程里，LLM 不是“想做什么就做什么”，而是被限制在当前 SOP 步骤内完成局部任务。

### 3.3.4 工具如何调用

项目中的 Agent 工具是后端封装的确定性函数，主要包括三个：

| 工具 | 入参 | 作用 | 使用阶段 |
| --- | --- | --- | --- |
| `search_stocks` | 公司简称、股票简称或股票代码 | 在 A 股股票列表中搜索候选标的 | 候选发现、候选补全 |
| `get_company_info` | 股票代码 | 获取公司基础信息，用于判断主营业务和供应链角色 | 业务确认 |
| `verify_stock_code` | 股票代码列表 | 校验最终候选代码是否真实存在 | 代码校验 |

工具调用采用 ReAct 文本协议，模型输出：

```text
Thought: 当前需要确认赣锋锂业是否属于锂电材料环节
Action: get_company_info
Action Input: SZ:002460
```

后端解析 `Action` 和 `Action Input` 后执行工具，并把结果作为 Observation 追加回模型上下文：

```text
Observation: {"code":"002460","info":{"股票简称":"赣锋锂业","行业":"能源金属",...}}
```

模型看到 Observation 后，再决定继续调用工具，或者输出当前步骤的 `Step Result`。

这里的关键点是：**Observation 只能由系统提供，模型不能自己编 Observation**。这可以防止模型伪造工具结果。

### 3.3.5 function call 风格的结构化输出设计

项目没有直接使用某个模型厂商强绑定的 function calling，而是设计了一个兼容性更好的“文本协议 + 后端解析”方案。原因是项目支持兼容 OpenAI 协议的不同模型，不同模型对原生 function calling 的支持程度不完全一致。

结构化输出分三类：

#### 1. 工具调用输出

```text
Thought: <说明为什么要调用工具>
Action: <工具名>
Action Input: <单行普通字符串参数>
```

后端解析后得到类似 function call 的结构：

```json
{
  "thought": "需要搜索宁德时代相关候选公司",
  "action": "search_stocks",
  "action_input": "宁德时代"
}
```

设计约束：

- `Action` 必须命中当前步骤允许的工具白名单。
- `Action Input` 只能是单行字符串。
- `search_stocks` 每次只能输入一个公司简称、股票简称或股票代码。
- 模型不能一次输出多个 Action。
- 模型不能自己输出 Observation。

这些约束把原本自由文本的工具调用变成了可解析、可执行、可拦截的结构化调用。

#### 2. 步骤结果输出

每个 SOP 步骤完成时，模型必须输出：

```text
Thought: <说明当前步骤完成依据>
Step Result:
{
  "summary": "本步骤结论",
  "candidate_stocks": []
}
```

不同步骤有不同的必填字段，例如：

| 步骤 | required_outputs |
| --- | --- |
| 候选发现 | `candidate_stocks` |
| 业务确认 | `confirmed_stocks` |
| 候选补全 | `expanded_candidates` |
| 结果分组 | `categories` |
| 代码校验 | `verified_codes` |
| 最终组装 | `theme` |

后端会检查 `Step Result` 是否是合法 JSON，以及必填字段是否存在且非空。缺字段就视为该步骤失败，并触发重试或返回错误。

#### 3. 最终答案输出

最终组装阶段必须输出：

```text
Final Answer:
{
  "id": "",
  "name": "宁德时代电池供应链",
  "description": "...",
  "categories": []
}
```

后端会把最终 JSON 构造成 `Theme` 模型，并检查最终答案中的股票代码是否全部来自 `verified_stock_codes`。

如果最终答案包含未验证代码，后端会拒绝该结果，并提示模型只使用已验证代码重新输出。

### 3.3.6 如何保证任务执行不偏

这个项目用了多层防偏机制，而不是只靠 prompt 约束。

#### 1. 固定 SOP 控制大方向

后端固定六个步骤，不让模型自由决定流程。模型不能在候选发现阶段直接输出最终答案，也不能在分组阶段重新调用搜索工具。

#### 2. 每个步骤有工具白名单

每个 `PlanStep` 都配置 `allowed_tools`：

- 候选发现只能用 `search_stocks`。
- 业务确认只能用 `get_company_info`。
- 结果分组不能调用工具。
- 最终组装不能调用工具。

如果模型输出了当前步骤不允许的工具，后端不会执行，而是提示模型改用允许工具或输出步骤结果。

#### 3. 每个步骤有最大行动次数

每个步骤都有 `max_actions`，例如候选发现最多 14 次、业务确认最多 12 次、代码校验最多 1 次。

这可以避免模型陷入无限搜索，也控制外部接口成本。

#### 4. 候选步骤必须真实调用搜索工具

候选发现和候选补全不能只凭模型常识直接输出候选。后端会检查是否实际调用过 `search_stocks`，否则拒绝该步骤结果。

#### 5. 候选覆盖不足会要求继续搜索

候选发现类步骤中，后端会统计工具 Observation 中收集到的候选代码数量。如果候选不足，并且行动额度还没用完，会要求模型继续使用单个公司简称调用 `search_stocks`，而不是过早输出 `Step Result`。

#### 6. 公司信息工具只能查已发现候选

业务确认阶段，`get_company_info` 的输入会被规范化。后端会优先要求模型使用已发现候选中的股票代码，避免模型凭空查一个未进入候选池的公司。

#### 7. 最终答案只能使用已验证代码

最终组装前，系统会调用 `verify_stock_code` 校验候选股票。最终 Theme JSON 中如果出现未验证代码，后端会拒绝输出。

#### 8. 格式错误自动修复

如果模型没有按 `Thought/Action/Action Input`、`Thought/Step Result` 或 `Final Answer` 格式输出，后端会追加修复提示，要求模型重新输出可解析格式。

#### 9. 检查点机制保证可恢复

执行状态会保存当前计划、当前步骤、已完成步骤、工具 Observation、已验证代码和最后错误。长任务失败或中断后，可以基于 checkpoint 恢复，而不是完全重跑。

### 3.3.7 这套设计解决了什么问题

| 问题 | 解决方式 |
| --- | --- |
| 大模型一次性生成容易幻觉 | 必须经过搜索、公司信息确认和代码校验 |
| ReAct 容易发散 | 全局 SOP 固定，步骤内才允许局部 ReAct |
| 工具调用不可控 | 每步工具白名单 + 最大行动次数 |
| 输出 JSON 不稳定 | Step Result / Final Answer 均由后端解析和校验 |
| 股票代码可能不存在 | 最终答案只能使用 `verify_stock_code` 通过的代码 |
| 过程不可见 | 每个步骤通过 SSE 推送 thinking、tool_call、tool_result、progress |
| 长任务中断难恢复 | checkpoint 保存执行状态 |

面试中可以把这部分总结成一句话：

> 我没有把 Agent 设计成完全自由的自动驾驶，而是设计成“系统给轨道，模型在轨道内做局部决策”。Plan-Execute 负责不跑偏，ReAct 负责处理局部不确定性，后端校验负责保证最终结果可用。

## 3.4 行情服务模块

行情服务负责把前端传入的股票代码转化为统一的 `StockQuote` 模型。

### 实时行情接口

项目当前使用雪球原始行情接口：

```text
https://stock.xueqiu.com/v5/stock/quote.json?symbol={SYMBOL}&extend=detail
```

例如：

```text
SZ002460
SH601138
BJxxxxxx
```

### 字段映射

| 原始字段 | 项目字段 | 说明 |
| --- | --- | --- |
| symbol/code | code | 统一格式化为 `SZ:002460` |
| name | name | 股票名称 |
| current | current_price | 现价 |
| percent | change_percent | 涨跌幅 |
| chg | change | 涨跌额 |
| last_close | prev_close | 昨收价 |
| open | open_price | 开盘价 |
| high | high | 最高价 |
| low | low | 最低价 |
| amount | volume | 成交额 |

### 关键工程问题

早期尝试直接使用 AkShare 的 `stock_individual_spot_xq`，但测试中出现 `KeyError: 'data'`，进一步排查发现雪球返回：

```json
{
  "error_code": "400016",
  "error_description": "遇到错误，请刷新页面或者重新登录帐号后再试"
}
```

这说明 AkShare 内置的雪球 token 或 Cookie 失效。最终方案是后端自己创建 `requests.Session()`，先访问雪球股票页获取 Cookie，再请求 quote JSON。

这个问题在面试中可以重点讲，因为它体现了数据源适配能力：不是简单调用第三方库，而是在第三方库不稳定时能够定位到底层 HTTP 交互并做可控封装。

## 3.5 K 线模块

K 线数据使用 AkShare 的腾讯日线接口封装：

```python
ak.stock_zh_a_hist_tx(symbol="sz002460", adjust="qfq")
```

项目固定取最近 22 个交易日，作为“近一个月日 K”。

### 为什么不是每次全量加载

主题下可能有几十只股票，如果每只股票都立即请求 K 线，会导致：

- 首屏加载慢。
- 后端线程池被占用。
- 外部行情接口压力变大。
- 本地配置、历史任务等轻接口也被拖慢。

因此项目采用两层优化：

1. 后端 K 线并发限制为 3。
2. 前端 K 线懒加载，图表进入视口附近才请求。

# 4. 性能优化复盘

## 4.1 问题一：切换主题时旧主题残留

### 现象

用户先选择“苹果手机供应链”，再切换到“宁德时代电池供应链”，页面仍显示苹果主题，直到新主题加载完成才切换。

### 根因

路由 ID 已经变化，但 Zustand 中的 `currentTheme` 仍然是旧主题。异步请求完成前，页面继续渲染旧状态。

### 解决方案

页面增加派生状态：只有 `currentTheme.id === routeId` 时才展示主题，否则显示“正在加载主题”。

### 复盘价值

这体现了前端异步状态管理中的一个典型问题：不能只看 Store 中有没有数据，还要看数据是否属于当前上下文。

## 4.2 问题二：页面切换后看板状态丢失

### 现象

用户切换到“AI分析”或“模型配置”后，再回到“供应链看板”，需要重新选择主题。

### 根因

导航中的看板入口固定跳转 `/dashboard`，没有携带当前主题 ID。

### 解决方案

如果当前存在主题，则看板导航跳转到 `/dashboard/{themeId}`。同时在创建主题后立即设置 `currentTheme`。

### 复盘价值

这说明“页面状态”和“URL 状态”需要同步。对于可分享、可恢复的页面，关键状态应该体现在路由中。

## 4.3 问题三：行情加载拖慢其他接口

### 现象

选择主题后，行情加载很慢；同时切换到模型配置或历史任务页面，本地文件接口也会变慢甚至失败。

### 根因

虽然模型配置和历史任务是本地文件读取，但它们仍运行在同一个 FastAPI 进程中。大量外部行情请求会占用线程池和请求处理资源，导致轻量接口排队。

### 解决方案

1. 不再全量下载 A 股行情。
2. 只按当前主题股票代码请求行情。
3. 实时行情并发限制为 5。
4. K 线并发限制为 3。
5. 实时行情设置 5 秒缓存。
6. K 线设置 1 小时缓存。
7. 前端 K 线懒加载。

### 复盘价值

这个问题体现了一个重要工程经验：即使某个接口本身很轻，只要它和重接口共用同一个服务进程，也可能被资源竞争拖慢。

## 4.4 问题四：后台日志被第三方库输出污染

### 现象

加载股票数据时，后台打印大量下载信息和进度条，影响业务日志可读性。

### 根因

AkShare 或底层数据源会向 stdout/stderr 输出下载进度。

### 解决方案

封装 `_run_quietly`，在调用第三方数据源时临时重定向 stdout/stderr，只保留项目自己的业务日志。

### 复盘价值

这体现了第三方库集成时的“可运维性”问题：功能可用不等于工程可用，日志可读性也是后端服务质量的一部分。

# 5. Bad Case 分析

## Bad Case 1：股票代码格式不一致导致前端显示 `--`

### 现象

后端明明请求到了行情，但前端现价和涨跌幅仍显示 `--`。

### 原因

前端使用主题中的股票代码作为 key，例如 `SZ:002460`。如果后端返回的行情代码是 `SZ002460` 或错误格式，`quotes[stock.code]` 就无法命中。

### 解决方案

后端统一用 `extract_numeric_code` 提取 6 位数字，再通过 `format_stock_code` 格式化为项目标准格式：

```text
SZ:002460
SH:601138
BJ:xxxxxx
```

### 经验

跨模块数据联动时，ID 或 key 的格式统一非常重要。显示层的 `--` 不一定代表接口没数据，也可能是 key 对不上。

## Bad Case 2：第三方封装接口不可用

### 现象

AkShare 封装的雪球接口报 `KeyError: 'data'`。

### 原因

底层雪球返回错误 JSON，而不是预期的 `data.quote`。

### 解决方案

绕过不稳定封装，直接请求雪球原始接口，并补上 Cookie 初始化步骤。

### 经验

第三方库不是黑盒。遇到数据异常时，要能够下钻到 HTTP 请求和原始响应层面排查。

## Bad Case 3：每个任务目录重复存股票基础列表

### 现象

`data/task_cache/` 下每个任务目录都有一份 `stock_list.json`。

### 原因

早期缓存路径按 task_id 隔离，导致全市场股票基础列表也被复制到任务目录下。

### 解决方案

把股票基础列表改成共享缓存，只保存到 `_shared/stock_list.json`。

### 经验

缓存要区分数据属性：

- 和任务强相关的数据适合任务级缓存。
- 全局稳定数据适合共享缓存。

# 6. 工程亮点

## 6.1 AI 与业务结构结合

项目不是简单调用大模型聊天，而是把大模型输出转成可被系统消费的结构化主题，包括分类、股票、描述和占比。这让 AI 结果可以进入后续业务流程。

## 6.2 流式交互提升体验

AI 分析通过 SSE 返回，用户可以看到分析过程，而不是等待一个长时间无反馈的 HTTP 请求。

## 6.3 行情数据源可控封装

项目没有把第三方数据接口直接暴露给业务层，而是通过 `akshare_adapter` 做统一适配，包括：

- 股票代码格式化。
- 字段映射。
- 异常兜底。
- 本地缓存。
- 静默第三方输出。

## 6.4 性能优化有明确抓手

从“全量行情下载”优化为“指定股票请求”，再配合缓存、并发限制和懒加载，解决了看板加载慢和轻接口被拖慢的问题。

## 6.5 状态管理贴近用户体验

看板主题、主题列表、行情数据分别管理，既避免重复分析，也避免页面切换丢状态。

# 7. 项目取舍

## 7.1 为什么用本地 JSON 而不是数据库

这个项目主要用于个人演示和面试复盘，本地 JSON 有三个优点：

1. 启动成本低，不需要额外数据库。
2. 数据结构直观，方便面试时展示。
3. 适合单用户、低并发、原型验证场景。

如果进入生产环境，需要迁移到数据库，并补充权限、事务、索引和备份能力。

## 7.2 为什么行情不直接保存在主题里

实时行情会不断变化，如果保存在主题里，会导致主题数据频繁更新，也会混淆“分析结构”和“实时数据”。

因此项目把主题和行情解耦：

- 主题保存供应链结构。
- 行情按需实时获取。
- K 线按需缓存。

## 7.3 为什么不用 WebSocket 刷新行情

当前行情刷新频率是 30 秒，属于低频轮询场景。用普通 HTTP 轮询更简单、稳定，也更适合演示项目。

如果后续要做秒级行情、盘口推送或多用户订阅，再考虑 WebSocket。

## 9.1 30 秒版本

这个项目是一个 AI 股票供应链分析大屏。用户输入一个产业主题，后端通过 ReAct Agent 调用大模型生成结构化供应链主题，包括分类、股票和业务说明。前端用 React 展示主题看板，并联动雪球实时行情和 AkShare 近一个月日 K。项目重点解决了 AI 输出结构化、页面状态保持、行情请求性能优化和第三方数据源适配问题。

## 9.2 2 分钟版本

我做的是一个“AI + 股票供应链 + 数据可视化”的全栈项目。用户在 AI 分析页输入主题，比如“宁德时代电池供应链”，系统会通过后端 ReAct Agent 调用大模型，把主题拆成供应链分类和相关 A 股公司，然后保存为本地主题。用户进入看板后，可以看到每个分类下的股票卡片、实时价格、涨跌幅和近一个月 K 线。

技术上，前端使用 React、TypeScript、Zustand 和 lightweight-charts，后端使用 FastAPI、SSE、Pydantic、AkShare 和雪球接口。比较有价值的地方是我把 AI 分析和实时行情解耦：主题结构是稳定数据，行情是动态数据。这样切换页面不会重新分析，停留在主题页时行情每 30 秒刷新。

项目过程中遇到过几个典型问题：一是切换主题时旧主题残留，我通过路由 ID 和当前主题 ID 匹配解决；二是行情加载拖慢本地配置接口，我发现原因是共用 FastAPI 线程池和进程资源，后来改成只请求当前主题股票，并加了并发限制、缓存和 K 线懒加载；三是 AkShare 封装的雪球接口失效，我下钻到原始 HTTP 响应，改为自己先获取 Cookie 再请求雪球 quote 接口。这个项目比较完整地体现了我在 AI 产品、前端交互、后端工程和数据源适配上的能力。

## 9.3 面试官可能追问

### 问：你这个项目和普通大模型问答有什么区别？

答：普通问答只生成文本，用户还要自己整理和使用。这个项目要求大模型输出结构化主题，包括分类、股票代码、公司名称和业务说明。结构化以后，系统才能把 AI 结果和行情接口、K线图、主题管理联动起来。所以它不是停留在“回答”，而是进入了后续业务流程。

### 问：为什么历史任务和配置接口会被行情加载影响？

答：因为它们虽然是本地文件读取，但和行情请求共用同一个 FastAPI 进程。行情请求会通过 `asyncio.to_thread` 调用同步第三方库，如果请求过多，会占用线程池和进程资源，导致其他轻量接口排队。后续我通过减少请求量、限制并发、增加缓存和 K线懒加载来解决。

### 问：为什么不用全量 A 股行情？

答：全量行情适合全市场扫描，但这个项目的场景是主题看板，只需要当前主题下几十只股票。全量下载会浪费网络和计算资源，还会拖慢页面。因此我改成按股票代码请求雪球单股行情。

### 问：为什么 K 线取 22 个交易日？

答：A 股一个自然月大约 20 到 22 个交易日，项目定位是看近一个月趋势，所以固定取最近 22 个交易日。这样前端展示稳定，也避免传入不同 count 导致看板不一致。

### 问：如果让你把它做成生产系统，你会怎么改？

答：我会从五个方面升级：第一，把本地 JSON 换成 PostgreSQL；第二，用 Redis 缓存行情和 K线；第三，用任务队列处理 AI 分析，避免长任务占用 API 请求；第四，增加用户体系和权限；第五，为 AI 输出增加来源引用、校验和人工确认，降低金融场景下的幻觉风险。

# 10. 总结

这个项目的核心价值不在于单个页面或单个接口，而在于把 AI Agent、产业链结构化分析、实时行情数据和前端可视化完整串起来。

从产品角度，它解决了主题研究信息分散、结构难沉淀、行情割裂的问题。

从技术角度，它覆盖了：

- React 前端状态管理和数据可视化。
- FastAPI 后端接口设计。
- SSE 流式 AI 分析。
- 第三方行情数据源适配。
- 缓存、并发控制和懒加载优化。
- 本地持久化和主题 CRUD。

从面试表达角度，它是一个比较完整的 AI 全栈项目，可以重点讲“为什么这么设计”“遇到什么问题”“如何定位和优化”，而不只是展示功能。
